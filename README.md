# KidOS Website

Static, self-contained website for `getkidos.com`, built to satisfy the four Google Play Console requirements the founder needed content for: developer account website, app website, support URL, and privacy policy. Three pages, one shared stylesheet, no build step.

## Files

- `index.html` — Home page. Serves as both the **Developer account website** and **App website** fields in Play Console; Google allows the same URL for both, and per `docs/strategy/07-website-strategy.md`'s explicit Phase 1 recommendation, this is a single lightweight landing page, not a multi-page marketing site — building more than this before Phase 2's content engine is running would be over-building relative to the roadmap.

  Sections, top to bottom: hero ("Who Your Child Becomes Starts Today"), a "What We Believe" belief-card marquee, problem (screen time / inconsistent routines / reward fatigue / parent burnout), thesis ("environment, not talent"), the five Daily Engine pillars ("Small Habits. Big Futures."), how it works (Daily Engine / Kid Wallet / Weekly Insights), a For Parents/For Kids benefits split ("Raise More Than Successful Kids," sourced from `docs/marketing/BRAND_MESSAGE_HOUSE.md`'s JTBD section), "Don't Just Manage Childhood. Shape It." (category differentiation), a trust strip, a closing Manifesto, and the waitlist. The page now bookends on philosophy — beliefs right after the hero, Manifesto right before the final CTA — with the concrete problem/product/benefits content carrying the middle.

## Deployment

Any static host works (Netlify, Vercel, GitHub Pages, Cloudflare Pages). No server or build step required — the three HTML files and `styles.css` can be uploaded as-is.

- **App website / Developer account website:** point `getkidos.com` at `index.html`.
- **Support URL:** the founder can use either `getkidos.com/support` (works out of the box with `support.html`, no DNS change needed) or `support.getkidos.com` as a subdomain — if the subdomain is preferred, add a DNS `CNAME` for `support` pointing at the same host and configure the host to serve `support.html` at that subdomain's root.
- **Privacy Policy:** `getkidos.com/privacy` — `privacy.html`.

## What's actually required for closed testing vs. what's optional

New personal Play Console accounts must run a closed test with at least 12 opted-in testers, continuously, for 14 days before applying for production access. Verified against current Play Console documentation (2026-07-08):

| Item | Required for closed testing? | Status |
|---|---|---|
| Support **email** | Yes — mandatory field, and required on the closed testing track (only the internal testing track is exempt from the Data Safety/privacy requirements) | Have it: `support.kidos@gmail.com` |
| **Privacy Policy URL** | Yes — required from the closed testing track onward, not just at production | Drafted in `privacy.html`, not yet legal-reviewed |
| Developer account website / App website | No — optional field | `index.html` exists, not a blocker either way |
| Support **URL** (a page, vs. just an email) | No — optional; the mandatory field is the support email, not a page | `support.html` exists, not a blocker either way |

Practical implication: you can start the 12-tester/14-day closed test as soon as the support email and privacy policy are in place — the homepage and support page are good practice for trust and eventual production conversion (and match `docs/strategy/07-website-strategy.md`'s own Phase 1 recommendation to have a minimal one), but they aren't gating your ability to begin testing.

## Before this goes live

This content was generated directly from the repository's existing brand and product documentation (`docs/marketing/BRAND_MESSAGE_HOUSE.md`, `docs/strategy/13-brand-guidelines.md`, `docs/product/REWARD_FLOW.md`, `docs/product/ONBOARDING_FLOW.md`), not written from a generic template — but several items still need real founder or legal action before publishing:

1. **`privacy.html` has not been reviewed by a lawyer.** As of the 2026-07-08 rewrite it no longer names COPPA or the DPDP Act on-page, and no longer shows internal "open item" callouts — but the underlying open questions weren't resolved, just moved off the page. They're tracked in `decisions/DECISION_LOG.md` and still unresolved: whether the current PIN-based parental consent flow meets COPPA's verifiable-consent bar (it likely doesn't, as designed), Google Play Families Policy ad-SDK configuration, and international data-transfer rules. India's DPDP Act's broader "child" definition (under 18, covering KidOS's entire audience) is a fact already reflected in how the policy is written, not an open item. None of the unresolved items are blockers for starting closed testing — they're checkable items for counsel before wider launch.
2. **Publisher name is now "Human Foundry"** (added 2026-07-08) — appears in `privacy.html`'s opening paragraph and in the footer copyright of all three pages (`© 2026 Human Foundry.`). This is a working/provisional name, not a registered company — COPPA's "operator" and DPDP's "Data Fiduciary" both explicitly cover individuals, so no registration is required to use it this way. Two things still open: (1) a quick domain/trademark/social-handle availability check before treating this as permanent — a search turned up an existing unrelated "Human Foundry" LinkedIn page, unclear what it does, worth a look before this is anywhere harder to change than a website footer; (2) **the Play Console "Developer name" field itself still needs to be set manually** in Play Console → Setup → Developer profile — this repo has no access to that console, so it's a founder action item, not something done here. Google allows a brand name that differs from the verified legal identity behind the account, as long as it doesn't misleadingly imply a registered company that doesn't exist (plain "Human Foundry," no "Inc."/"LLC"/"Pvt Ltd." suffix, is fine).
3. **Contact email is live: `support.kidos@gmail.com`.** Used consistently across all three pages for now — good enough for MVP and closed testing. Once getkidos.com email is set up, consider forwarding `support@getkidos.com` to this same inbox rather than switching, so a branded address exists without needing full email hosting.
4. **The waitlist form on `index.html`** uses a `mailto:` fallback, which opens the visitor's email client rather than submitting silently — functional for a Phase 0/1 placeholder, but worth replacing with a real form backend (Mailchimp, a simple serverless function, etc.) once the waitlist is actually being promoted.
5. **This is the website, not the Play Store listing itself.** `docs/strategy/06-playstore-strategy.md` is the strategic reasoning for the listing (title, screenshots, ASO); the actual listing copy (short/full description, keywords) is a separate, still-unwritten deliverable — the `Playstore/` folder at the repo root remains the natural home for it once it's drafted.
